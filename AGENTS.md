# AGENTS INSTRUCTIONS

## Project Overview

This is **garyinnerarity.com** — a static portfolio and knowledge-base site for Gary Innerarity, Solutions Engineer & MBA. It is hosted on GitHub Pages with the custom domain `garyinnerarity.com`. There is no build framework (no React, no Vite, no bundler) — just plain HTML, CSS, and vanilla JS served directly.

### Key Pages

- `index.html` — Main portfolio site (hero, work, stack, credentials, notes viewer, skills viewer, timeline)
- `index-brain.html` — Interactive 3D "Cognitive Architecture" brain visualization (Three.js)

### Content Directories

| Directory  | Purpose                                                                          | Registry                |
| ---------- | -------------------------------------------------------------------------------- | ----------------------- |
| `notes/`   | Markdown knowledge-base articles rendered client-side via marked.js              | `notes/file-list.json`  |
| `skills/`  | Markdown technical skill write-ups rendered client-side via marked.js            | `skills/file-list.json` |
| `assets/`  | Images, certificates, global assets                                              | —                       |
| `scripts/` | JS modules loaded by `index.html` (particles, visitor counter, quotes, certs)    | —                       |

## Mandatory Workflows

### 1. Repository Health — Always Validate Before Committing

Before any commit or when asked to review the repo, **always** run the repo health scanner skill:

```bash
bash .agents/skills/repo-health-scanner/scan.sh
```

Address any critical or high-severity findings before committing. Security vulnerabilities and lint errors must be resolved, not deferred.

### 2. Sitemap & robots.txt — Keep Updated

After any content change (new pages, new markdown files, updated sections), regenerate the sitemap:

```bash
npm run build
```

This runs three scripts in sequence:

1. `node notes/generate-file-list.js` — scans `notes/` for `.md` files → writes `notes/file-list.json`
2. `node skills/generate-file-list.js` — scans `skills/` for `.md` files → writes `skills/file-list.json`
3. `node generate-sitemap.js` — reads both file-list JSONs → writes `sitemap.xml`

**Rules:**

- After adding or removing any `.md` file in `notes/` or `skills/`, run `npm run build` to regenerate the file lists and sitemap.
- If a new top-level HTML page is added, add it to the `staticUrls` array in `generate-sitemap.js`.
- `robots.txt` allows all crawlers (search engines and AI). If a new page or section is added that should be excluded, update `robots.txt` with a `Disallow:` rule. Otherwise, leave it as-is.
- The sitemap URL in `robots.txt` must always point to `https://garyinnerarity.com/sitemap.xml`.

### 3. JSON Schema Integrity

#### `events.json`

Timeline events displayed on the main site. Each entry must follow this schema exactly:

```json
{
  "date": "YYYY-MM-DD",
  "title": "Short title",
  "description": "Brief description of the event.",
  "type": "project | training | certificate | health"
}
```

- `date`: ISO 8601 date string (YYYY-MM-DD), required
- `title`: Short display title, required
- `type`: One of `"project"`, `"training"`, `"certificate"`, or `"health"` — required, determines timeline icon/color
- `description`: One-sentence description, required
- Array must remain valid JSON. Entries are sorted chronologically by the site.

#### `notes/file-list.json` and `skills/file-list.json`

Auto-generated arrays of relative markdown file paths. **Never edit these manually** — they are generated by the `generate-file-list.js` scripts. Run `npm run build` to regenerate.

#### `assets/particlesjs-config.json`

Particles.js configuration. Edit only when changing the visual particle effect on the main page.

### 4. Markdown File Standards

All `.md` files in `notes/` and `skills/` are rendered client-side by marked.js with the following conventions:

#### Structure

- **H1 title required**: Every file must start with a single `# Title` on line 1. This is used as the display name in the sidebar and outline pane.
- **No YAML frontmatter**: The site does not parse frontmatter. Do not add `---` blocks.
- **Wikilinks supported**: Use `[[Note Name]]` to link between notes. The site resolves these by matching against filenames (case-insensitive, `.md` extension stripped). For example, `[[Projects]]` resolves to `projects.md`, `[[About]]` resolves to `about.md`.

#### Formatting

- Use ATX headings (`#`, `##`, `###`). The outline pane is generated from these.
- Fenced code blocks with language tags (` ```python `, ` ```bash `, ` ```json `, etc.) are syntax-highlighted by highlight.js.
- Mermaid diagrams are supported inside ` ```mermaid ` blocks.
- Standard Markdown for lists, bold, italic, links, and images.

#### File Organization

- Top-level `.md` files in `notes/` or `skills/` go in the "General" sidebar group.
- Subdirectory names become sidebar group names — use lowercase with spaces (e.g., `architecture and ops/`, `hosting models/`, `platform and orchestration/`).
- `Welcome.md` in `notes/` is always sorted first in the sidebar.

#### Naming Conventions

- Use lowercase filenames with spaces allowed (the site handles them): `cloud-providers.md`, `home-lab.md`.
- Subdirectory names use lowercase with spaces: `architecture and ops/`, `hosting models/`.
- Keep names concise and descriptive — they appear directly in the sidebar.

### 5. CSS Standards

- Both `styles.css` (main site) and inline `<style>` in `index-brain.html` use CSS custom properties defined in `:root`.
- Always include the standard property alongside vendor prefixes:

  ```css
  -webkit-background-clip: text;
  background-clip: text;
  ```

- No CSS preprocessors — plain CSS only.

### 6. JavaScript Standards

- Vanilla JS only — no TypeScript, no JSX, no module bundler.
- Three.js is loaded from CDN (`r128`) for the brain visualization.
- marked.js, highlight.js, mermaid.js, and particles.js are loaded from CDN.
- ESLint configuration is provided in `.eslintrc.json` — run `npx eslint .` to check for issues.
- Scripts in `scripts/` are module-like files loaded by `index.html`.

### 7. Asset Management

- Optimize images before committing. Prefer JPEG for photos/textures, PNG only when transparency is needed.
- Keep individual image files under 1 MB when possible.
- Certificate badge images live in `assets/certs/`.
- GitHub assets live in `assets/github/`.
- Global/shared assets live in `assets/globals/`.
- `.DS_Store` files are gitignored — never commit them.

## Deployment Pipeline

The site is deployed via GitHub Actions (`.github/workflows/static.yml`). On push to `main`:

1. Checkout → Setup Node 20 → `npm run build` (regenerate file lists + sitemap)
2. **Dev file cleanup** — reads `.pagesignore` and removes matched files/dirs from the workspace before upload
3. Upload remaining files as a Pages artifact → Deploy to GitHub Pages

### `.pagesignore`

Lists files and directories to exclude from the deployed site. One pattern per line; blank lines and `#` comments are ignored. Patterns are expanded as shell globs and passed to `rm -rf`.

Update `.pagesignore` when adding new dev-only files (configs, tooling, editor dirs) that should not be served on the live site.

## llms.txt

The file `llms.txt` at the repository root is an AI-readable summary of the site owner's identity, expertise, certifications, and projects. Update it when:

- New certifications are earned
- New major projects are completed
- Job title or expertise areas change

Keep it factual, structured, and aligned with the content on the live site.

### AI Context — Mandatory Update Rule

Whenever a new project, certification, skill, or milestone is added **anywhere** on the site (timeline events, work cards, credentials, etc.), the following **must** also be updated in the same change:

1. **`#ai-context` section in `index.html`** — the visually-hidden, machine-readable summary near the bottom of the page. Add a paragraph or subsection describing the new addition.
2. **`llms.txt`** — the AI-readable summary at the repo root. Add the new item to the appropriate section (Enterprise Projects, Personal Projects, Certifications, etc.).

This ensures LLM crawlers and AI indexing systems always have a complete, up-to-date picture of all projects and credentials. **Never add content to the visible site without also updating both AI context sources.**
